
## 一、核心认知

> **LangChain 1.0 的 Agent 本质是：一个“可持久化的状态机”**

不是简单的“模型调用 + messages 拼接”。



## 二、关键概念一览

| 概念            | 本质                      |
| ------------- | ----------------------- |
| state         | Agent 的完整运行现场（大脑快照）     |
| invoke        | 推动状态机向前运行一步             |
| thread_id     | 状态机的会话编号                |
| checkpointer  | state 的存储系统（记忆硬盘）       |
| InMemorySaver | 内存版 checkpointer（不持久化）  |
| SqliteSaver   | SQLite 持久化 checkpointer |
| Middleware    | 在状态机生命周期关键节点插入钩子        |



## 三、state 里包含什么？

state ≠ 仅 messages

典型包含：

```
state = {  
messages, # 对话历史  
intermediate_steps, # 工具调用轨迹  
tool_outputs,  
next_action,  
其他 LangGraph 内部字段  
}
```

> state = Agent 的完整运行现场



## 四、invoke 的真实行为

```python
agent.invoke(input, config={"thread_id": "1"})
```

真实过程：

```
① 从 checkpointer 读取 state
② Middleware 介入
③ 模型推理 / 工具调用
④ 生成新的 state
⑤ 覆盖保存到 checkpointer
```

> invoke 是“状态推进”，不是“无状态调用模型”。



## 五、checkpointer 的职责

只做一件事：

> 根据 thread_id 存取 state

它**不知道**：

- 对话内容
- 是否摘要
- 是否被修改

它只是：

```
存什么 → 下次就读什么
```



## 六、InMemorySaver vs SqliteSaver

| 类             | 存储        | 持久化 | 场景    |
| ------------- | --------- | --- | ----- |
| InMemorySaver | 内存        | ❌   | 学习/测试 |
| SqliteSaver   | SQLite 文件 | ✅   | 生产环境  |

Agent 行为完全一致，只是 state 存储位置不同。



## 七、thread_id 的作用

用于区分不同 state：

```
checkpointer = {
  "A": state_A,
  "B": state_B
}
```

> thread_id 决定你加载哪一个“Agent 大脑”。



## 八、Middleware 的作用位置

插入在：

|钩子|时机|
| --- | - |
|before_model|模型调用前|
|after_model|模型调用后|
|before_tool|工具调用前|
|after_tool|工具调用后|

Middleware **可以直接修改 state**。



## 九、SummarizationMiddleware 的本质

> 不是“做摘要”  
> 是 **重写 state.messages 的结构**

触发时机：

```
before_model(state)
```

做的事情：

```
旧 messages（50轮） → 摘要 SystemMessage
state.messages = [摘要, 最近几条]
```



## 十、最关键事实（很多人误解）

### ❗摘要不是“逻辑忽略历史”
### ❗而是“物理删除历史”

旧消息：

> **真的从 state 中消失**



## 十一、invoke 时 state 的读写机制（极其关键）

### 1️⃣ 从 checkpointer 读取

```
读取 → 反序列化 → 得到内存副本
```

不是原地修改。



### 2️⃣ Middleware / 模型 修改的是副本

包括：

- 重写 messages
- 新增 AIMessage
- 修改轨迹



### 3️⃣ invoke 结束时

```
checkpointer[thread_id] = 新 state
```

**整块覆盖（overwrite）**

不是 append。



## 十二、Summarization 对 checkpointer 的真实影响

| 影响       | 结果   |
| -------- | ---- |
| state 体积 | 永久变小 |
| 恢复速度     | 更快   |
| token 消耗 | 长期降低 |
| 对话轮数     | 理论无限 |

> checkpointer 中存的，不再是完整历史，而是“压缩后的记忆形态”。



## 十三、正确的认知模型

以后理解 Agent 时，应当这样想：

```
checkpointer 存的不是“聊天记录”
而是“Agent 当前大脑快照”
```

每次 invoke：

```
读取大脑 → 运行 → 保存新的大脑
```

SummarizationMiddleware：

```
在运行过程中，重塑大脑的记忆结构
```



## 十四、一句话总结全链路

> **Agent 是状态机**  
> **state 是大脑**  
> **checkpointer 是硬盘**  
> **thread_id 是大脑编号**  
> **invoke 是大脑运行一次**  
> **Middleware 可以改大脑**  
> **SummarizationMiddleware 会重构大脑记忆**
