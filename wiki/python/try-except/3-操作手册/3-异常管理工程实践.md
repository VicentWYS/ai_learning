# 异常管理工程实践 —— 从单点处理到体系化管理

> 前面我们学会了"在单个函数里怎么写 try-except"。但实际项目中可能有几十个函数、上百处调用点——异常处理需要一个系统级的管理方案。

---

## 一、问题的升级：从"单个函数"到"整个项目"

当你只有一个 `main.py` 时，异常管理很简单。但实际项目可能是：

```
my_ai_app/
├── api/              # Web 接口层（FastAPI/Flask）
│   ├── routes.py     # 路由处理
│   └── middleware.py # 中间件
├── services/         # 业务逻辑层
│   ├── embedding.py  # 向量化服务
│   ├── chatbot.py    # 对话服务
│   └── document.py   # 文档处理服务
├── clients/          # 外部 API 调用层
│   ├── llm_client.py # LLM API 封装
│   └── vector_db.py  # 向量数据库封装
├── utils/
│   └── exceptions.py # ← 所有自定义异常定义在这里
├── config.py
└── main.py
```

**如果每个函数各自写一遍完整的 try-except，会出现三个问题：**

1. **重复代码泛滥**：每个 API 调用都写同样的重试逻辑
2. **处理不一致**：同一个异常在 A 处重试 3 次，在 B 处只重试 1 次
3. **排查困难**：报错散落在各处，没有统一的入口查看

---

## 二、三个层级的异常管理体系

```
┌────────────────────────────────────────────┐
│  层级 1：基础设施层                          │
│  负责：重试、超时、熔断等通用能力              │
│  工具：装饰器、上下文管理器、retry 库          │
└──────────────┬─────────────────────────────┘
               │
┌──────────────▼─────────────────────────────┐
│  层级 2：服务层                              │
│  负责：把底层异常翻译成业务异常                │
│  工具：自定义异常树、异常转换                  │
└──────────────┬─────────────────────────────┘
               │
┌──────────────▼─────────────────────────────┐
│  层级 3：接入层                              │
│  负责：统一兜底、格式化输出、通知告警           │
│  工具：全局异常钩子、中间件、监控平台           │
└────────────────────────────────────────────┘
```

## 三、层级 1：基础设施层——把通用能力抽成可复用组件

### 3.1 装饰器：消除重复的 try-except

最常见的重复代码就是"调用 API → 超时重试 → 打日志"。用装饰器一次定义，到处使用：

```python
import time
import logging
import functools
from typing import Type, Tuple

logger = logging.getLogger(__name__)


def retry_on_transient(
    max_retries: int = 3,
    backoff_base: float = 2.0,
    retryable_exceptions: Tuple[Type[Exception], ...] = (TimeoutError, ConnectionError),
):
    """
    装饰器：对临时性异常自动重试。

    用法（只需一行）：
        @retry_on_transient(max_retries=3, retryable_exceptions=(TimeoutError,))
        def call_llm(prompt):
            return model.invoke(prompt)
    """
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            last_error = None
            for attempt in range(1, max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except retryable_exceptions as e:
                    last_error = e
                    if attempt == max_retries:
                        raise  # 重试耗尽，向上传播
                    wait = backoff_base ** attempt
                    logger.warning(
                        f"{func.__name__} 失败 (第 {attempt}/{max_retries} 次): "
                        f"{type(e).__name__}. {wait:.0f}s 后重试..."
                    )
                    time.sleep(wait)
            # 理论上不会到这里（最后一次 attempt 会 raise）
            raise last_error  # type: ignore
        return wrapper
    return decorator


# ── 使用示例 ────────────────────────────────────────────
# 之前：每个函数写一大段重试逻辑
# 之后：一行装饰器搞定

@retry_on_transient(max_retries=3)
def call_embedding_api(text: str) -> list[float]:
    """调用 Embedding API。超时或连接失败自动重试 3 次。"""
    return embedding_client.embed(text)


@retry_on_transient(max_retries=2, retryable_exceptions=(TimeoutError,))
def call_chat_api(messages: list) -> str:
    """调用 Chat API，只对超时重试（连接失败不重试）。"""
    return chat_client.chat(messages)
```

### 3.2 和成熟库对比：什么时候自己写，什么时候用现成的

| 方案 | 适用场景 | 示例 |
|------|----------|------|
| 自己写装饰器 | 逻辑简单、想完全控制行为 | 上面那个 `@retry_on_transient` |
| `tenacity` 库 | 需要高级重试策略（随机抖动、条件重试、异步支持） | `@tenacity.retry(wait=tenacity.wait_exponential())` |
| `backoff` 库 | 轻量级需求，只想要退避重试 | `@backoff.on_exception(backoff.expo, TimeoutError, max_tries=3)` |

**建议：初学者从自己写装饰器开始（理解原理），项目变大后切换到 `tenacity`。**

```python
# tenacity 示例（成熟方案）
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10),
    retry=retry_if_exception_type((TimeoutError, ConnectionError)),
)
def call_llm(prompt: str):
    return model.invoke(prompt)
```

## 四、层级 2：服务层——异常翻译 + 业务语义

底层 API 抛出的异常是技术语言（`openai.AuthenticationError`），但上层不关心你用的是 OpenAI 还是 DeepSeek。服务层的职责是**把技术异常翻译成业务异常**。

### 4.1 异常翻译模式

```python
# exceptions.py —— 集中管理所有自定义异常
class AppError(Exception):
    """应用根异常——所有自定义异常的基类"""
    def __init__(self, message: str, user_message: str | None = None):
        super().__init__(message)
        self.user_message = user_message or message  # 给用户看的


class LLMServiceError(AppError):
    """LLM 服务相关异常的总称"""
    pass


class LLMQuotaExhausted(LLMServiceError):
    """额度用完了"""
    pass


class LLMServiceUnavailable(LLMServiceError):
    """服务不可用（重试后仍失败）"""
    pass


class EmbeddingError(AppError):
    """向量化相关异常"""
    pass


class ConfigInvalid(AppError):
    """配置无效"""
    pass
```

### 4.2 在调用层翻译异常

```python
# clients/llm_client.py —— 这里是唯一直接接触 openai SDK 的地方

import openai

class LLMClient:
    """LLM API 的封装层。所有对 openai SDK 的调用都在这里。"""

    def chat(self, messages: list) -> str:
        try:
            response = self._raw_call(messages)
            return response

        except openai.AuthenticationError as e:
            # 把 SDK 异常 → 翻译成业务异常
            raise ConfigInvalid(
                message=f"LLM API Key 无效: {e}",
                user_message="API Key 配置无效，请检查 .env 文件。"
            ) from e

        except openai.RateLimitError as e:
            if "insufficient_quota" in str(e):
                raise LLMQuotaExhausted(
                    message=f"LLM 额度用尽: {e}",
                    user_message="API 额度已用完，请充值或更换 Key。"
                ) from e
            raise LLMServiceUnavailable(
                message=f"LLM 限流: {e}",
                user_message="请求太频繁，请稍后重试。"
            ) from e

        except (openai.APITimeoutError, openai.APIConnectionError) as e:
            raise LLMServiceUnavailable(
                message=f"LLM 服务连接/超时: {e}",
                user_message="AI 服务暂时不可用，请稍后重试。"
            ) from e

        except openai.APIError as e:
            raise LLMServiceError(
                message=f"LLM API 错误: {e}",
                user_message="AI 服务异常，请稍后重试。"
            ) from e

    @retry_on_transient(max_retries=3)
    def _raw_call(self, messages: list) -> str:
        """实际的 API 调用，带重试装饰器。"""
        # 这里抛出的是 openai 的原始异常
        # 外层 chat() 方法负责翻译
        completion = self.client.chat.completions.create(
            model=self.model_name,
            messages=messages,
        )
        return completion.choices[0].message.content
```

### 为什么这个模式很重要？

```
之前（异常直达上层）：
  api/routes.py
      ↓ 直接收到 openai.AuthenticationError
      ↓ routes.py 需要 import openai，知道 openai 的异常体系

之后（异常翻译）：
  api/routes.py
      ↓ 收到 ConfigInvalid（业务语言）
      ↓ routes.py 只需要 import 自己的 exceptions.py
  clients/llm_client.py
      ↓ 唯一 import openai 的地方
```

**效果：如果将来把 OpenAI 换成 DeepSeek 或 Anthropic，你只需要改 `clients/llm_client.py` 这一个文件，上层代码完全不受影响。**

## 五、层级 3：接入层——最后的防线

不管前面几层处理得多好，总会有漏网之鱼。接入层的工作是**确保用户永远不会看到裸的 traceback**。

### 5.1 Web 应用：全局异常中间件（FastAPI 示例）

```python
# api/middleware.py
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from my_app.utils.exceptions import (
    AppError,
    ConfigInvalid,
    LLMQuotaExhausted,
    LLMServiceUnavailable,
)

app = FastAPI()


@app.exception_handler(AppError)
async def app_error_handler(request: Request, exc: AppError):
    """捕获所有自定义异常，返回结构化 JSON"""
    # 根据异常类型映射 HTTP 状态码
    status_map = {
        ConfigInvalid: 400,           # 客户端配置问题
        LLMQuotaExhausted: 429,       # 额度用尽
        LLMServiceUnavailable: 503,   # 服务不可用
    }
    status_code = status_map.get(type(exc), 500)

    return JSONResponse(
        status_code=status_code,
        content={
            "error": type(exc).__name__,
            "message": exc.user_message,    # ← 给用户看的
            # exc.message 只写日志，不返回给用户
        }
    )


@app.exception_handler(Exception)
async def unhandled_exception_handler(request: Request, exc: Exception):
    """终极兜底：所有未被前面捕获的异常"""
    logger.exception(f"未处理的异常: {request.url}")  # 完整栈写入日志

    return JSONResponse(
        status_code=500,
        content={
            "error": "InternalError",
            "message": "服务器内部错误，我们已记录此问题，请稍后重试。"
            # 不暴露任何技术细节！
        }
    )
```

### 5.2 命令行/脚本应用：全局异常钩子

```python
# main.py
import sys
import logging

logger = logging.getLogger(__name__)


def global_exception_hook(exc_type, exc_value, exc_traceback):
    """
    终极兜底：捕获所有未被 try-except 处理的异常。
    设置这个后，即使你忘了写 try-except，用户也不会看到裸 traceback。
    """
    # KeyboardInterrupt 是用户主动中断，正常退出即可
    if issubclass(exc_type, KeyboardInterrupt):
        sys.__excepthook__(exc_type, exc_value, exc_traceback)
        return

    # 所有其他未捕获的异常
    logger.critical("未捕获的异常，程序将退出", exc_info=(exc_type, exc_value, exc_traceback))

    print(f"\n❌ 程序遇到未预期的错误而退出。")
    print(f"   错误类型：{exc_type.__name__}")
    print(f"   详细信息已记录到日志文件。")


# 安装全局钩子（在程序入口最前面）
sys.excepthook = global_exception_hook


# ── 你的主程序 ──────────────────────────────────────────

def main():
    # 即使这里没有任何 try-except，
    # 如果崩溃了，用户看到的也是上面那个友好的提示
    result = some_ai_pipeline()
    print(result)


if __name__ == "__main__":
    main()
```

### 5.3 异常监控平台

开发阶段靠日志，生产环境靠监控。当异常发生时，你需要**主动知道**，而不是等用户来告诉你：

| 工具 | 类型 | 适用场景 |
|------|------|----------|
| **Sentry** | 异常监控 SaaS | 自动收集报错、聚合去重、通知告警。个人项目免费额度够用 |
| **Loguru** | Python 日志库 | 比标准库 logging 更好用的日志方案 |
| **OpenTelemetry** | 可观测性标准 | 链路追踪 + 日志 + 指标，适合较复杂的系统 |

```python
# Sentry 集成示例（基本不需要改代码）
import sentry_sdk

sentry_sdk.init(
    dsn="https://xxx@sentry.io/xxx",  # 从 Sentry 后台获取
    traces_sample_rate=0.1,            # 采样10%的请求做性能追踪
)

# 之后所有未捕获的异常会自动上报到 Sentry
# 你会在邮件/钉钉/飞书上收到通知，附带完整的调用栈和变量值
```

## 六、完整的项目异常管理清单

把前面三层串起来，形成一个可操作的清单：

### 基础设施层（一次配置，全项目使用）

- [ ] 有一个**重试装饰器**（自己写或用 `tenacity`）
- [ ] 所有外部 API 调用都加了重试装饰器
- [ ] 关键配置项（重试次数、超时时间）从配置文件读取，不是硬编码

### 服务层（每个外部依赖对应一个翻译层）

- [ ] 有一个 `utils/exceptions.py` 集中定义所有自定义异常
- [ ] 自定义异常继承了统一的基类（`AppError`）
- [ ] 自定义异常同时包含 `message`（技术细节）和 `user_message`（用户提示）
- [ ] 每个外部依赖（LLM、向量库、数据库）有自己的 client 封装类
- [ ] client 封装类负责把外部 SDK 异常翻译成自定义异常

### 接入层（最后一道防线）

- [ ] Web 应用：有全局异常中间件，返回结构化 JSON
- [ ] 命令行脚本：安装了 `sys.excepthook`
- [ ] 生产环境：接入了异常监控（至少 Sentry 免费版）
- [ ] 用户永远不会看到裸的 traceback

## 七、一张图总结三层体系

```
用户请求
    │
    ▼
┌──────────────────────────────────────────────┐
│  接入层（FastAPI middleware / sys.excepthook） │
│  ┌────────────────────────────────────────┐  │
│  │ 任何逃逸的异常在这里被兜底               │  │
│  │ → 返回用户友好的结构化错误              │  │
│  │ → 上报 Sentry / 写日志                 │  │
│  └────────────────────────────────────────┘  │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│  服务层（exceptions.py + client translation）  │
│  ┌────────────────────────────────────────┐  │
│  │ openai.AuthenticationError             │  │
│  │   → ConfigInvalid("请检查API Key")      │  │
│  │                                        │  │
│  │ openai.RateLimitError                  │  │
│  │   → LLMQuotaExhausted("额度用尽")       │  │
│  └────────────────────────────────────────┘  │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│  基础设施层（@retry 装饰器 / tenacity）        │
│  ┌────────────────────────────────────────┐  │
│  │ TimeoutError   → 等 2s → 等 4s → 等 8s  │  │
│  │ ConnectionError → 等 2s → 等 4s → 等 8s  │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

**这三个层级，你不需要一次性全部搭建。开发路径建议：**

1. **第一天**：只管写业务代码，手动 try-except，报错就加（前面两篇文档的方法）
2. **第一周**：发现重复了，把重试逻辑抽成装饰器（层级 1）
3. **第一个月**：发现上层代码依赖底层 SDK 异常，开始建自定义异常 + 翻译（层级 2）
4. **上线前**：加全局中间件和 Sentry（层级 3）

**不要一开始就建三层体系，被问题推着走，什么时候发现痛点什么时候加结构。**
