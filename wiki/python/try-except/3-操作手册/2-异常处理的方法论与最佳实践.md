# 异常处理的方法论与最佳实践

> 从"能 catch 住不崩"到"有体系地设计异常处理"。

---

## 一、核心方法论：按"响应策略"分类，而非按异常类型

初学者容易陷入一个误区：见到一种异常就加一个 except，最终写出这样的代码：

```python
# ❌ 按类型罗列——没有逻辑，只是照抄报错
try:
    response = model.invoke(prompt)
except openai.AuthenticationError as e:
    ...  # 跟下面这个处理方式完全一样
except openai.PermissionDeniedError as e:
    ...  # 跟上面那个处理方式完全一样
except openai.RateLimitError as e:
    ...
except openai.APITimeoutError as e:
    ...
except openai.APIConnectionError as e:
    ...
```

**正确思路：先把异常按"你应该怎么做"分组，再写代码。**

### 分类框架

所有异常，按**响应策略**可以分为四类：

```
                可恢复
                  │
     ┌────────────┼────────────┐
     │            │            │
  重试即可    用户改了就行    只能打日志
 (网络抖动)   (配置错误)    (未知bug)
     │            │            │
     ▼            ▼            ▼
 自动重试     提示用户     记录+优雅降级
     │            │            │
     └────────────┼────────────┘
                  │
              不可恢复 → 安全退出 + 通知用户
```

### 四种策略的映射关系

| 策略 | 含义 | 典型异常 | 代码行为 |
|------|------|----------|----------|
| **A. 重试** | 临时性故障，等一会儿再试大概率能好 | `TimeoutError`、`ConnectionError`（偶发）、`RateLimitError` | 自动重试 N 次，指数退避 |
| **B. 提示用户修正** | 用户侧的配置/输入有问题，改了就能好 | `ValueError`、`AuthenticationError` | 打印明确指引 + 停止当前操作 |
| **C. 记录+降级** | 不该发生但发生了，不影响核心流程 | 某个非关键步骤的未知异常 | 打日志 + 返回默认值或跳过 |
| **D. 安全退出** | 无法继续，继续运行会出更大问题 | 初始化失败、内存不足、数据损坏 | 记录完整现场 + 通知用户 + 退出 |

**方法：拿到一个异常，先问自己它是 ABCD 中的哪一类，再决定代码怎么写。**

## 二、AI 开发中完整的异常处理代码示例

假设你有一个调用 LLM 的函数，可能遇到的异常有七八种。下面展示如何按 ABCD 分类来组织代码：

```python
import time
import logging
from openai import (
    AuthenticationError,
    RateLimitError,
    APITimeoutError,
    APIConnectionError,
    APIError,
)

logger = logging.getLogger(__name__)
MAX_RETRIES = 3

def call_llm_with_error_handling(model, prompt: str) -> str | None:
    """
    调用 LLM，内建完整的异常处理体系。
    返回值：成功时返回模型回复文本；不可恢复时返回 None。
    """

    # ── A 类异常：可重试 ──────────────────────────────────
    RETRYABLE = (APITimeoutError, APIConnectionError, RateLimitError)

    for attempt in range(1, MAX_RETRIES + 1):
        try:
            response = model.invoke(prompt)
            return response.content

        # ── A：重试即可 ──────────────────────────────────
        except RETRYABLE as e:
            wait = 2 ** attempt  # 指数退避：2s → 4s → 8s
            logger.warning(
                f"第 {attempt}/{MAX_RETRIES} 次调用失败 ({type(e).__name__})，"
                f"{wait}s 后重试..."
            )
            if attempt == MAX_RETRIES:
                logger.error(f"重试 {MAX_RETRIES} 次仍失败，放弃。")
                print(f"❌ 服务暂时不可用，请稍后重试。")
                return None
            time.sleep(wait)

        # ── B：用户修正即可，不重试 ────────────────────────
        except AuthenticationError as e:
            print(f"🔑 API Key 无效，请检查 .env 文件中的 DEEPSEEK_API_KEY。"
                  f"\n详细信息：{e}")
            return None

        # ── C/D 判断：是已知的 API 错误还是完全未知？ ─────
        except APIError as e:
            # APIError 是 OpenAI SDK 所有 API 错误的基类
            # 我们不知道具体是什么，但知道跟 API 调用有关
            logger.error(f"API 错误：{type(e).__name__}: {e}")
            print(f"❌ API 返回错误，请联系服务商或稍后重试。")
            return None

        except Exception as e:
            # 完全未知的错误——D 类
            logger.exception("调用 LLM 时发生未预期的异常")
            print(f"❌ 发生了意料之外的错误，详情已记录日志。")
            return None
```

### 这个代码背后的设计决策

| 决策 | 为什么这么做 |
|------|-------------|
| 重试次数用 `MAX_RETRIES = 3` | 太少没用，太多等太久。3 次是比较通用的经验值 |
| 指数退避 `2 ** attempt` | 给服务器喘息时间，第 3 次重试间隔 8 秒足够大多数瞬时故障恢复 |
| 重试耗尽后返回 `None` 而非抛异常 | 调用方可以检查返回值做降级处理，而不是自己也被迫处理异常 |
| `AuthenticationError` 不重试 | 重试 N 次也不会让错误的 Key 变正确，立即失败更节省时间 |
| `except Exception` 用 `logger.exception()` | 自动附带完整调用栈，方便事后排查 |

## 三、三个在实战中验证过的最佳实践

### 实践 1：永远不要写空的 except 块

```python
# ❌ 我可能也犯过这个错
try:
    response = model.invoke(prompt)
except Exception:
    pass  # 错误被默默吞掉，程序继续跑，你永远不知道发生了什么

# ✅ 至少打一行日志
try:
    response = model.invoke(prompt)
except Exception as e:
    logger.warning(f"调用失败，跳过：{e}")
    response = None
```

**"程序不崩"不等于"没问题"。** 被你吞掉的异常，将来会变成更难排查的 bug。

### 实践 2：异常消息要区分"给用户看的"和"给开发者看的"

```python
except ConnectionError as e:
    # 给用户看：通俗易懂，告诉 ta 该做什么
    print(f"🌐 无法连接 API 服务器，请检查网络或 Base URL 配置。")

    # 给开发者看：技术细节，用于排查
    logger.error(f"ConnectionError: {e}", exc_info=True)
```

| 对象 | 信息要求 | 示例 |
|------|----------|------|
| 给用户 | 通俗 + 可操作 | "API Key 无效，请检查 .env 文件" |
| 给开发者 | 精确 + 完整 | `openai.AuthenticationError: 401 - ...` + 调用栈 |

### 实践 3：自定义异常体系——让你自己的代码也能被"分类捕获"

当项目变大，你自己的代码也可能在不同层级抛出异常。建立一个简单的自定义异常树：

```python
# ── 自定义异常体系 ──────────────────────────────

class AIEngineeringError(Exception):
    """本项目所有异常的基类"""
    pass

class ConfigError(AIEngineeringError):
    """配置类错误 —— B 类：用户修正即可"""
    pass

class LLMCallError(AIEngineeringError):
    """LLM 调用类错误"""
    pass

class LLMRetryableError(LLMCallError):
    """可重试的 LLM 错误 —— A 类"""
    pass

class LLMFatalError(LLMCallError):
    """不可恢复的 LLM 错误 —— D 类"""
    pass


# ── 使用自定义异常 ──────────────────────────────

def load_config():
    api_key = os.getenv("DEEPSEEK_API_KEY")
    if not api_key or api_key == "your_api_key_here":
        raise ConfigError("DEEPSEEK_API_KEY 未设置或仍为占位符")
    return api_key

def main():
    try:
        config = load_config()
        result = call_llm(config)
    except ConfigError as e:
        # B 类：告诉用户改配置
        print(f"⚙️ 配置问题：{e}")
    except LLMRetryableError as e:
        # A 类：自动重试
        ...
    except LLMFatalError as e:
        # D 类：记录 + 退出
        logger.exception("致命错误")
        print(f"❌ 程序终止：{e}")
```

这样做的好处是：**上层调用者不需要知道底层是 `ValueError` 还是 `KeyError`，它只需要知道这是 `ConfigError`——属于 B 类策略。**

## 四、决策流程图：拿到一个异常，怎么处理？

```
新的异常发生了
      │
      ▼
  ┌─────────────────────────┐
  │ 问：重试一次大概率能好吗？  │
  │ (网络超时、限流、连接断开)  │
  └──────────┬──────────────┘
             │
        ┌────┴────┐
       是          否
        │           │
        ▼           ▼
   ┌────────┐  ┌──────────────────────────┐
   │ A 类    │  │ 问：用户改了配置/输入就能好？│
   │ 自动重试 │  │ (Key 错误、参数非法、额度不足) │
   │ 指数退避 │  └──────────┬───────────────┘
   └────────┘             │
                     ┌────┴────┐
                    是          否
                     │           │
                     ▼           ▼
                ┌────────┐  ┌──────────────────────────┐
                │ B 类    │  │ 问：发生在这里会阻塞核心流程吗？│
                │ 提示用户 │  │                          │
                │ 明确指引 │  └──────────┬───────────────┘
                └────────┘             │
                                  ┌────┴────┐
                                 是          否
                                  │           │
                                  ▼           ▼
                             ┌────────┐  ┌────────┐
                             │ D 类    │  │ C 类    │
                             │ 安全退出 │  │ 记录+降级│
                             │ 保留现场 │  │ 跳过/默认│
                             └────────┘  └────────┘
```

### 一个边界情况的处理

有时候同一类异常在不同场景下属于不同策略：

```python
# 场景 1：加载用户头像（非核心功能）
try:
    avatar = download_image(url)
except ConnectionError:
    logger.warning("头像加载失败，使用默认头像")   # ← C 类：降级
    avatar = DEFAULT_AVATAR

# 场景 2：加载核心模型配置
try:
    model_config = download_config(config_url)
except ConnectionError:
    raise RuntimeError("无法加载模型配置，程序终止") from e  # ← D 类：安全退出
```

**同一个 `ConnectionError`，在非核心场景属于 C 类，在核心场景属于 D 类。分类取决于业务重要性，不是异常类型本身。**

## 五、一个检查清单

写完异常处理代码后，逐条自查：

- [ ] 每个 `except` 块都有具体的异常类型（没有裸 `except:` 或裸 `except Exception:` 吞错）
- [ ] 可重试的异常（超时、限流、连接抖动）有重试逻辑，不可重试的（认证失败、参数错误）没有浪费重试
- [ ] 给用户的提示信息是"通俗 + 可操作"的，不只是把英文错误原样输出
- [ ] 给开发者的技术细节通过 `logging` 输出，不直接打印到用户屏幕上
- [ ] 空的 `pass` 或直接 `return` 的 except 块至少有一行日志
- [ ] 有兜底的 `except Exception` 且带有 `logging.exception()` 或 `traceback`
- [ ] 关键操作失败后，程序不会带着"半残"状态继续运行

---

## 六、总结

| 层次 | 问什么 | 得到什么 |
|------|--------|----------|
| 方法论 | 这个异常属于 ABCD 哪一类？ | 决定处理策略 |
| 代码结构 | 按策略分组，不要按类型罗列 | 代码清晰、好维护 |
| 最佳实践 | 不吞异常、区分用户/开发者信息、自定义异常树 | 生产级质量 |
| 检查清单 | 写完后逐条对照 | 查漏补缺 |

**一句话：不要做"异常类型的收集者"，要做"异常策略的设计者"。** 先分类、再编码，这才是从入门到熟练的分水岭。
