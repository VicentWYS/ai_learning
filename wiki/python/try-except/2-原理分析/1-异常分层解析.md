# 多层异常处理详解 —— 以 `4-错误处理.py` 为例

> 面向 AI 开发初学者，逐层拆解代码中的异常处理设计思路。

---

## 一、代码中的两层异常处理

打开 [4-错误处理.py](4-错误处理.py)，可以看到异常处理分布在**两个层级**：

### 第一层：业务函数内部（`example_4_error_handling`）

```python
def example_4_error_handling():
    try:
        response = model.invoke("请用一句话介绍什么是智能体")
        print(f"\nAI回复：\n{response.content}")

    except ValueError as e:          # ← 捕获特定异常
        print(f"配置错误：{e}")

    except ConnectionError as e:     # ← 捕获特定异常
        print(f"网络错误：{e}")

    except Exception as e:           # ← 兜底捕获
        print(f"未知错误：{type(e).__name__}: {e}")
```

### 第二层：主函数（`main`）

```python
def main():
    try:
        example_4_error_handling()
    except Exception as e:
        print(f"\n运行出错：{e}")
        import traceback
        traceback.print_exc()        # ← 打印完整调用栈
```

---

## 二、为什么需要多层异常处理？

你可以把异常处理想象成**医院的急诊分诊**：

| 层级 | 类比 | 职责 |
|------|------|------|
| 第一层 `example_4_error_handling` | 科室医生 | 针对**已知的、可预期的**问题做"专科处理"——给用户友好提示 |
| 第二层 `main` | 急诊总值班 | 兜底：任何第一层没挡住或者没想到的问题，在这里做"总控处理"——记录完整日志 |

### 具体来说：

**第一层（业务函数内部）** 的职责是：
- **你大概知道哪里会出错**，比如调用模型时，可能是网络不通、API Key 配错了
- 对不同的错误给出**有针对性的提示**（"配置错误" vs "网络错误"），用户一看就知道该修什么
- 让程序**继续运行**而不是直接崩溃

**第二层（main 函数）** 的职责是：
- 捕获那些**你没预料到的错误**（比如 model.invoke 返回了奇怪的数据格式）
- 用 `traceback.print_exc()` 打印**完整调用栈**，方便事后排查
- 防止整个程序**直接崩掉**——至少给用户留一句交代

## 三、`except` 的匹配顺序：先具体，后宽泛

这是 Python 异常处理最核心的原则：

```python
except ValueError as e:       # 1️⃣ 最具体
    ...
except ConnectionError as e:  # 2️⃣ 次具体
    ...
except Exception as e:        # 3️⃣ 最宽泛（兜底）
    ...
```

Python 会**从上到下**依次匹配异常类型，**第一个匹配到的 except 块被执行，后面的全部跳过**。

### 如果把顺序写反了会怎样？

```python
# ❌ 错误写法
except Exception as e:        # 这个会吃掉所有异常
    ...
except ValueError as e:       # 这个永远不会被执行！
    ...
```

因为 `ValueError` 是 `Exception` 的子类，它已经被上面的 `except Exception` 拦截了。这就是为什么要**先具体、后宽泛**。

### 异常继承关系（简化版）：

```
BaseException
 └── Exception
      ├── ValueError          ← 数值/配置类错误
      ├── ConnectionError     ← 网络连接类错误
      │    ├── ConnectionRefusedError
      │    └── ConnectionResetError
      ├── TimeoutError        ← 超时
      ├── RuntimeError        ← 运行时错误
      └── ...（还有很多）
```

## 四、实际开发中如何设计异常处理？

作为 AI 开发初学者，你可以遵循以下步骤：

### 步骤 1：先写核心逻辑，不加 try-except

```python
def call_llm(prompt):
    response = model.invoke(prompt)
    return response.content
```

先让代码跑起来，**实际遇到什么错误，再针对性添加处理**。不要一开始就"过度防御"。

### 步骤 2：运行中观察常见错误，逐类添加处理

运行几次后，你会发现常见错误无非这几类：

| 异常类型 | 典型场景 | 处理策略 |
|----------|----------|----------|
| `ValueError` | API Key 没配、参数不合法 | 提示用户检查配置 |
| `ConnectionError` | 网络不通、Base URL 写错 | 提示用户检查网络 |
| `TimeoutError` | 模型响应太慢 | 可以选择重试 |
| `RateLimitError`（部分 SDK 有） | 调用频率超限 | 等待后重试 |

```python
try:
    response = model.invoke(prompt)
except ValueError as e:
    print(f"⚠️ 配置有问题：{e}")        # 用户看
except ConnectionError as e:
    print(f"🌐 网络连不上：{e}")        # 用户看
except Exception as e:
    print(f"❌ 出了个意料之外的问题：{e}")  # 兜底
    # 这里可以把完整错误记到日志文件
```

### 步骤 3：在程序入口加一层"终极兜底"

```python
def main():
    try:
        # 你的所有业务逻辑
        my_ai_app()
    except Exception as e:
        # 最后一道防线：别让程序直接崩
        print(f"程序异常退出：{e}")
        import traceback
        traceback.print_exc()  # 开发阶段，打印全栈方便排查
```

### 步骤 4：生产环境中把 `traceback` 换成日志系统

```python
import logging

logging.basicConfig(
    filename='app.log',
    level=logging.ERROR,
    format='%(asctime)s - %(levelname)s - %(message)s'
)

def main():
    try:
        my_ai_app()
    except Exception as e:
        print(f"程序遇到问题，详情已记录到日志文件。")
        logging.error("未捕获的异常", exc_info=True)  # 写入文件而非打印到屏幕
```

## 五、总结：记住这三句话

1. **先具体、后宽泛** —— except 的顺序决定了匹配优先级
2. **内层做"专科处理"，外层做"兜底保护"** —— 每层各司其职
3. **不要提前过度设计** —— 先让代码跑起来，遇到什么错误再加什么处理

对于 AI 开发而言，调用大模型 API 是最容易出错的地方（网络、配额、参数），所以至少在这类调用外面包一层 `try-except`。其他的地方，等实际踩坑了再加。
